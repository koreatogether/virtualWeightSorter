# 🌱 아두이노 초보자를 위한 복잡한 프로젝트 접근 가이드

> **"처음부터 완벽한 코드는 없다. 끊임없이 기록하고 수정하며 완성해 나가는 것이다."**
> 
> 이 가이드는 `4채널 온도 모니터링 시스템` 개발 과정(`progressLog.md`)에서 얻은 교훈을 바탕으로, 아두이노 초보자가 복잡한 프로젝트를 시작할 때 어떻게 접근해야 하는지를 설명합니다.

---

## 🚀 1단계: 작게 시작하고 하나씩 검증하라 (Divide & Conquer)
초보자가 가장 많이 하는 실수는 **"센서 연결하고, LCD 연결하고, 코드를 한 번에 짜서 실행하는 것"**입니다. 99% 확률로 실패합니다.

### ✅ 우리의 경험 (progressLog 참조)
우리는 이 프로젝트를 한 번에 만들지 않았습니다.
1. `01` 버전: 화면에 그래프가 나오는지 확인
2. `02` 버전: 하드웨어 시리얼 통신 테스트
3. `Probe` 테스트: 센서만 따로 떼어내어 주소와 값을 확인
4. `08` 버전: 최종 통합

**💡 Tip:**
- **센서 테스트**: 기능을 구현하기 전에 센서 예제를 먼저 돌려서 값이 정확한지, 순서는 맞는지 확인하세요. (우리는 이것을 안 해서 S2 센서가 안 나오는 문제를 겪었습니다!)
- **화면 테스트**: 센서 없이 가짜 숫자(Dummy Data)를 넣어서 화면이 예쁘게 나오는지 먼저 확인하세요.

---

## 💾 2단계: 하드웨어의 한계를 존중하라 (Memory Limits)
컴퓨터와 달리 아두이노(특히 Nano/Uno)는 메모리(RAM)가 매우 적습니다. (겨우 2KB!)

### ✅ 우리의 경험
- 라이브러리를 추가하자 메모리 사용량이 80%를 넘었습니다.
- **해결책**: 욕심을 버려야 합니다. 
    - `BUFFER_LENGTH`를 120에서 100으로 줄여서 공간을 확보했습니다.
    - 불필요한 전역 변수를 삭제했습니다.

**💡 Tip:**
- 컴파일할 때 나오는 **"스케치가 프로그램 저장 공간의 X%를 사용함"** 문구를 무시하지 마세요.
- RAM이 80%를 넘어가면 이상한 오류(화면 멈춤, 값 깨짐)가 발생할 수 있습니다.

---

## ⏱️ 3단계: 시간의 마법사가 되어라 (No delay)
`delay(1000)`은 1초 동안 아두이노를 **'기절'**시키는 명령어입니다. 화면 터치도, 센서 읽기도 못 합니다.

### ✅ 우리의 경험
- 처음엔 화면이 뚝뚝 끊겼습니다.
- **해결책**: `millis()` 함수를 사용해 **"1초가 지났니?"** 라고 시계만 쳐다보며 다른 일을 계속 하게 만들었습니다(Non-blocking). 덕분에 그래프를 그리면서도 터치 버튼이 즉각 반응합니다.

---

## 📝 4단계: 기록은 최고의 스승이다 (Log Everything)
개발하다 보면 어제 무슨 코드를 고쳤는지 기억이 안 납니다.

### ✅ 우리의 경험
- `progressLog.md`에 날짜별로 **문제(증상) -> 원인 -> 해결책**을 기록했습니다.
- "갑자기 S2 센서가 안 되네?" -> 로그를 보고 "아, 아까 시뮬레이션 코드 지울 때 실수했구나"라고 바로 찾을 수 있었습니다.

---

## 🏁 마치며
이 프로젝트도 처음엔 단순히 "온도를 재보자"에서 시작했습니다.
하지만 **센서가 꼬이고(순서 문제)**, **메모리가 터지고(RAM 부족)**, **화면이 안 나오는(프로토콜 충돌)** 수많은 문제를 겪으며 `Ver 08`까지 왔습니다.

실패는 과정일 뿐입니다. 로그를 남기고, 하나씩 해결해 보세요. 당신도 할 수 있습니다! 🛠️
