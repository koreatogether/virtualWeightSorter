# ADR-[Number]: [Title]

* **상태**: [제안됨 / 승인됨 / 완료됨 / 대체됨]
* **날짜**: YYYY-MM-DD
* **작성자**: [이름]

## 1. 맥락 (Context)
이 결정을 내리게 된 배경과 해결하려는 문제입니다. 어떤 기술적 제약이나 사용자 요구사항이 있었는지 기술합니다.

## 2. 결정 (Decision)
우리가 선택한 해결책과 그 이유를 설명합니다. 다른 대안이 있었다면 왜 이 방법이 최선이었는지도 포함하면 좋습니다.

## 3. 결과 (Consequences)
이 결정으로 인해 얻게 되는 이점 혹은 감수해야 할 트레이드오프(단점)를 기술합니다.

---

### 💡 좋은 ADR 작성을 위한 팁
1. **간결성**: 너무 길게 적기보다 핵심 논리 위주로 작성합니다.
2. **불변성**: 한 번 승인된 ADR은 수정하지 않고, 내용이 바뀌면 새로운 ADR(번호 증가)을 작성하여 이전 ADR을 '대체(Supersede)' 상태로 바꿉니다.
3. **공유**: 팀원 혹은 나중에 이 코드를 볼 미래의 나에게 설명한다는 느낌으로 작성합니다.

---

### ✅ 작성 예시 (Example)

# ADR-0001: 버튼 기반 화면 전환 도입

* **상태**: 승인됨
* **날짜**: 2026-01-20
* **작성자**: GitHub Copilot

## 1. 맥락 (Context)
현재 시스템은 5초마다 자동으로 화면이 전환됩니다. 하지만 사용자가 특정 데이터를 오래 확인하고 싶을 때 강제로 화면이 바뀌어 불편하다는 피드백이 있었습니다.

## 2. 결정 (Decision)
D3 핀에 물리 버튼을 추가하고, 이 버튼을 누를 때마다 `currentDisplayMode` 변수를 토글하는 방식을 도입하기로 했습니다. 자동 전환 기능은 유지하되, 버튼 입력 시 타이머를 리셋하여 사용자 조작 우선순위를 높입니다.

## 3. 결과 (Consequences)
* **장점**: 사용자가 원하는 정보를 원하는 만큼 고정해서 볼 수 있음.
* **단점**: 추가 배선(D3)이 필요하며, 버튼 디바운싱(Debouncing) 처리를 위한 코드가 추가됨.
